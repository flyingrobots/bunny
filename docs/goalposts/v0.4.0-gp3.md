# Goalpost v0.4.0-GP3: Compression Decoders

## Description

Define and implement a deterministic decoder for a canonical Bunny compressed
mesh byte stream inside `bunny-codec`.

## Status

Active - design contract first.

## Problem

GP1 introduced stable quantized mesh buffers in `bunny-mesh`. GP2 introduced
file format adapters in `bunny-codec`. GP3 closes the mesh commons track by
adding a decoder for a Bunny-native compressed byte profile that reconstructs
validated mesh buffers without depending on external file format semantics.

## Non-Goals

* Do not add new OBJ, PLY, glTF, STL, or other external format profiles.
* Do not add compression encoders in this goalpost.
* Do not add transform, animation, material, texture, or scene graph semantics.
* Do not weaken existing GP2 zero-copy parser claims.
* Do not claim zero-copy or zero-allocation decoder behavior until the byte
  profile and API shape make that contract provable.
* Do not mark GP3 complete until the implementation, tests, docs, and CI evidence
  support every acceptance criterion below.

## Canonical Byte Profile

The GP3 decoder profile is intentionally Bunny-native and must decode into
existing `bunny-mesh` concepts rather than inventing another mesh model.

The profile must define:

* Magic bytes and version field.
* Endianness for every integer field.
* Header length and total payload length validation.
* Vertex count and triangle count limits.
* Quantization bounds needed to reconstruct `QuantizedMesh16`-compatible
  vertices.
* Vertex payload coding.
* Triangle index payload coding.
* Optional checksum or verification field, if implemented.
* Canonical error variants for malformed bytes, unsupported versions, overflow,
  invalid counts, invalid bounds, invalid indices, and trailing bytes.

## Design Invariants

* Decoding must be deterministic across native and WASM targets.
* Accepted-path decoding must not panic.
* Malformed input must return explicit errors.
* Integer arithmetic must use checked or saturating operations according to the
  documented error policy.
* The decoded representation must validate index bounds before exposing a mesh
  view.
* The accepted path allocation contract must be explicit before implementation.
* If the decoder needs owned output buffers, the API and tests must name that
  requirement instead of implying borrowed-view behavior.
* Fixture bytes must be canonical and checked into the repository.

## Acceptance Criteria

* A documented Bunny compressed mesh byte profile exists in this goalpost.
* `bunny-codec` exposes a decoder API for the canonical profile.
* The decoder returns a validated view or owned decoded buffers according to the
  allocation contract recorded before implementation begins.
* The decoder rejects unsupported versions, invalid lengths, invalid counts,
  invalid quantization bounds, invalid indices, non-canonical payload encodings,
  integer overflow, and trailing bytes.
* Native tests cover the accepted path, each public error variant, and at least
  one fixture corpus case.
* WASM tests cover the accepted path and the canonical malformed-input corpus.
* Native allocation witnesses prove the accepted-path allocation contract, even
  if that contract intentionally permits bounded output allocation.
* `CHANGELOG.md`, `ROADMAP.md`, and this goalpost document record the final
  evidence before GP3 is marked completed.

## Slice Plan

### Slice 3.1: Decoder Profile Contract

Status: Active

Define the byte layout, error model, fixture corpus shape, and verification
requirements. This slice may update docs only. It must not mark GP3 completed.

### Slice 3.2: Decoder Implementation

Status: Planned

Implement the public decoder API and typed borrowed view in `bunny-codec` against
the Slice 3.1 contract.

### Slice 3.3: Determinism And Evidence Gate

Status: Planned

Add native and WASM corpus tests, malformed-input regressions, allocation
witnesses, docs, and changelog evidence. GP3 can be marked completed only after
this gate is green.

## Open Design Questions

* Should the canonical profile use delta-coded quantized vertices, varint-coded
  index deltas, or fixed-width packed sections first?
* Should the profile include a checksum in v0.4.0, or should content addressing
  remain the responsibility of `bunny-mesh` hashes?
* Should the borrowed view expose compressed byte sections only, or should it
  expose typed decoded records through iterators like `ObjMesh`?
* What exact maximum counts should the decoder enforce before arithmetic and
  slice offset calculation?
